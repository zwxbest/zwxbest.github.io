    javac的source和target的含义？

    • -source 指定用哪个版本的编译器对java源码进行编译

    • -target 指定生成的class文件将保证和哪个版本的虚拟机进行兼容。我们可以通过-target 1.2来保证生成的class文件能在1.2虚拟机上进行运行，但是1.1的虚拟机就不能保证了。
      jdk8默认的source和target都是1.8，但是用了maven可能默认是1.5，需要在pom中做一下配置

      怎么看常量池中有多少个常量？

      从偏移地址0X08开始的u2大小，即0809字节，随便打开一个是0032，那就是50，表示49个，从1开始，1-49.

      在什么情况下可能会出现内存泄漏？

      内存泄漏是指无用对象（不再使用的对象）持续占有内存或无用对象的内存得不到及时释放，从而造成内存空间的浪费称为内存泄漏。

    　　长生命周期的对象持有短生命周期对象的引用就很可能发生内存泄漏，尽管短生命周期对象已经不再需要，但是因为长生命周期持有它的引用而导致不能被回收，这就是Java中内存泄漏的发生场景。

    造成内存泄漏的几种情况：

    静态集合类引起内存泄漏

    　　像HashMap、Vector等的使用最容易出现内存泄露，这些静态变量的生命周期和应用程序一致，他们所引用的所有的对象Object也不能被释放，因为他们也将一直被Vector等引用着。

    当集合里面的对象属性被修改后，再调用remove()方法时不起作用。

    监听器

    　　在释放对象的时候却没有去删除这些监听器，增加了内存泄漏的机会。

    各种连接

    　　比如数据库连接（dataSourse.getConnection()），网络连接(socket)和io连接，除非其显式的调用了其close（）方法将其连接关闭，否则是不会自动被GC 回收的。

    内部类和外部模块的引用

    　　内部类的引用是比较容易遗忘的一种，而且一旦没释放可能导致一系列的后继类对象没有释放。此外程序员还要小心外部模块不经意的引用，例如程序员A 负责A 模块，调用了B 模块的一个方法如： public void registerMsg(Object b); 这种调用就要非常小心了，传入了一个对象，很可能模块B就保持了对该对象的引用，这时候就需要注意模块B 是否提供相应的操作去除引用。

    单例模式

    　　不正确使用单例模式是引起内存泄漏的一个常见问题，单例对象在初始化后将在JVM的整个生命周期中存在（以静态变量的方式），如果单例对象持有外部的引用，那么这个对象将不能被JVM正常回收，导致内存泄漏。

    如何处理堆溢出

    简单思路

    • 虚拟机的堆参数(-Xmx与 -Xms〉，与机器物理内存对比看是否还可以调大
    • 是否存在某些对象生命周斯过长
    • 持有状态时间过长的情况，尝试减少程序运行期的内存消耗

    引用计数是否可以判断对象是否存活？

    无法解决循环引用的问题

    可达性分析

    从根往下走，走过的路径称为引用链。到GC ROOTS没有引用链，就是不可达。

    引用分类

    强引用

    强 引 用 就 是 指 在 程 序 代 码 之 中 普 遍 存 在 类 似 -“ Object.obj = new Objedt()” 这类的
    引用 ，只要 强引用还存在，•垃圾 收集器永远不会回收棹被引用的对象。

    软引用

    软引用用来描述一些还有用但并非必需的对象。对于软引用关联着的对象，在系统将发生内存溢出异常之前回收。如果这次回收还没有足够的内存。才会抛出内存溢出异常。在JDKL2之活，提供了softReference类来实现软引用

    弱引用

    弱引用也是用来描述非必需対象的，佴是它的强度比软引用更弱一些，被弱引用关联的对象只能生存到下一次垃圾收集发生之前。垃圾回收器工作作时，无论当前内存是否足够，都会回收掉只被弱引用关联的対象。

    虚引用

    唯一的作用是被回收时收到一个通知

    你了解哪几种垃圾回收算法？

    标记-清除算法

    分为标记和清除两步

    缺点

    效率，空间碎片，不连续

    复制算法

    空间分为两块，将存活的复制到另一块。
    将内存分为较大的Eden和两块相同的survior空间，大小8：1，将存活的复制到另一块survivor空间，空间不够提前进入老年代

    标记-整理

    标记后整理挪到一边

    分代收集

    新生代用复制算法，老年代用标记-整理算法

    HotSpot的收集算法

    用OopMap存放对象引用，减少停顿，只有到安全点，Safepoint才进行GC，各个线程主动轮询中断标志
    但是如果碰到线程没有分配CPU时间，不执行的时候，比如Sleep或者Blocked状态。这时引入安全区域，安全区域是指引用关系不会发生变化的区域，在区域中任何地方GC都是安全的。发起GC时，不用管Safe Region状态的线程。如果要离开，检查是否完成GC过程，否则等待到GC过程完成

    Parallel Scavenge

    目的是达到一个可控制的吞吐量，吞吐量就是运行用户时间与总时间的比例。两个参数，:MaxGCPauseMillis，收集器尽量保证回收时间不超过这个数。，GC停顿时问缩短是以牺牲吞吐量和新生代空间来换取的：
    系统把新生代调小一些，收集300MB新生代背定比收集500MB快吧，这也直接导致垃圾
    收集发生得更频繁一些，原来10秒收集一次、每次停顿100毫秒，现在变成5秒收集一次、
    毎次停顿70毫秒。停顿时间的确在下降，但吞吐量也降下来了。

    GCTimeRatio参数的值应当是一个大于0且小于100的整数，也就是垃圾收集时间占总
    吋间的比率，相当于是吞吐ft的倒数。如果把此参数设为19,那允许的最大GC时间就占
    总时间的5%(即1/(1+19))，默认值为99,就是允许最大1%(即1/(1+99))的垃圾收
    集时间。

    由于吞吐量关系密切，ParallelScavenge收集器也经常称为“吞吐量优先”收集器。
    除L:述两个参数之外，ParallelScavenge收集路还有一个参数-XX:+UseAdaptiveSizePolicy
    值得关注。这是一个开关参数，当这个参数打开之后，就不需要手工指定新生代的大
    小(-Xmn)、Eden与Survivor区的比例(-XX:SurvivorRatio)、晋升老年代对象年龄
    (-XX:PretcnureSizeThreshold)等细节参数了，虚拟机会根据当前系统的运行情况收集性能监80第二部分自动内存管理机制
    控信息，动态凋整这些参数以提供最合适的停顿时间或者最大的吞吐址，这种调节方式称为
    GCA适应的凋节策略（GCErgonomics）

    Serial Old 收集器

    SerialOld是Serial收集器的老年代版木，它同样是一个单线程收®器，使用“标记一
    整理”算法。这个收集器的主要意义也是在于给Client模式下的虚拟机使用，如果在Server
    校式下，那么它主要还有两大用途：一种用途是在JDK1.5以及之前的版本中与Parallel
    Scavenge收集器搭配使用t另一种用途就是作为CMS收集器的后备预案，在并发收集发牛
    ConcurrentModeFailure时使用。

    Parallel Old 收集器

    ParallelOld是ParallelScavenge收集器的老年代版本，使用多线程和“标记一整理”算
    法。直到ParallelOld收集器出现后，“吞吐泉优宄”收集器终十有了比较名副其实的应用组
    合，在注重吞吐以及CPU资源敏感的场合，都卩f以优先考虑ParallelScavenge加Parallel
    Old收集器

    CMS收集器

    ，CMS收集器是基于“标记二-淸除”算，分为初始标记、并发标记、重新标记、并发清除四步
    法实现的，:它的运作过程相对于前曲儿种收集器米说更复杂一些，，初始标 4己、重 新 标 记 这 两 个 步 骤 仍 然 要 “ Stop The World”。
    由于 整个 过程 巾耗 时最 长的 井发 标记 和并 发清 除过 程收 集器 线程 都可 以与 用户 线程 -起
    I.作 ，所以 ，从总体上 来说 ，C M S收集器的内存回收过程是与用户线程一起并发执行的。
